# AutoResearch Engine Design

**Date:** 2026-03-15
**Status:** Approved
**Crate:** `hive_learn`

## Summary

An automated prompt improvement loop for Hive's internal skill system. Inspired by autoresearch principles (Andrej Karpathy), the engine iterates on skill prompts by running binary eval suites, measuring pass rates, mutating underperforming prompts via AI, and keeping winners â€” all autonomously with configurable guardrails.

## Problem

Hive's `PromptEvolver` can version and rollback prompts, and `SelfEvaluator` can measure aggregate quality trends, but there is no automated closed-loop system that:

1. Defines objective, per-skill eval criteria
2. Runs those evals repeatedly to get statistically meaningful pass rates
3. Mutates prompts based on specific failure evidence
4. Compares candidates and promotes winners
5. Repeats until convergence or budget exhaustion

Users must manually notice quality problems, manually rewrite prompts, and manually verify improvements. The autoresearch engine closes this loop.

## Three Ingredients

1. **Objective metric:** Binary eval pass rate (0.0-1.0) computed from yes/no questions per skill
2. **Automated measurement:** `EvalRunner` executes the skill and judges output against eval criteria
3. **Something to change:** The skill's `prompt_template` text, versioned by `PromptEvolver`

## Approach

New `AutoResearchEngine` in `hive_learn` that orchestrates existing components (`PromptEvolver`, `LearningStorage`, `SkillMarketplace::scan_for_injection`) plus new focused structs (`EvalSuite`, `EvalRunner`, `PromptMutator`). Each piece is small, independently testable, and composed by the engine into the improvement loop.

Alternatives considered:
- **Extend PromptEvolver directly:** Rejected â€” makes PromptEvolver a god struct
- **Agent-based via HiveMind:** Rejected â€” over-engineered for a tight loop

## Component Architecture

```
+---------------------------------------------------+
|              AutoResearchEngine                    |
|  (orchestrates the loop, owns the config)          |
|                                                     |
|  run() -> AutoResearchReport                       |
|  run_single_iteration() -> IterationResult         |
|  should_continue() -> bool                         |
+---------+---------+---------+---------------------+
| EvalSuite| EvalRunner| Prompt  | PromptEvolver     |
|          |           | Mutator | (existing)        |
| defines  | executes  | rewrites| versions &        |
| criteria | evals     | prompts | rollback          |
+----------+-----------+---------+-------------------+
```

## Data Types

### EvalSuite

```rust
pub struct EvalSuite {
    pub skill_name: String,
    pub questions: Vec<EvalQuestion>,
    pub source: EvalSource,
}

pub struct EvalQuestion {
    pub id: String,                  // e.g. "valid_rust_code"
    pub question: String,            // "Does the output contain valid Rust code?"
    pub weight: f64,                 // default 1.0
}

pub enum EvalSource {
    Explicit,       // from TOML [[eval]] section
    AutoGenerated,  // AI-derived from skill description
    Hybrid,         // mix of both
}
```

Eval suites are hybrid: skills can define explicit `[[eval]]` questions in TOML, and missing evals are auto-generated from the skill description via AI. Explicit questions always take priority.

### Eval Results

```rust
pub struct EvalResult {
    pub question_id: String,
    pub passed: bool,
    pub reasoning: String,           // AI's one-line justification
}

pub struct EvalRunResult {
    pub pass_rate: f64,              // 0.0-1.0
    pub results: Vec<EvalResult>,
    pub sample_outputs: Vec<String>, // actual skill outputs evaluated
}
```

### Iteration Tracking

```rust
pub struct IterationResult {
    pub iteration: u32,
    pub prompt_text: String,
    pub eval_result: EvalRunResult,
    pub is_new_best: bool,
    pub improvement_over_baseline: f64,
}

pub struct AutoResearchReport {
    pub skill_name: String,
    pub iterations_run: u32,
    pub baseline_pass_rate: f64,
    pub final_pass_rate: f64,
    pub best_prompt_version: u32,
    pub improvement: f64,
    pub stopped_reason: StopReason,
    pub iteration_history: Vec<IterationResult>,
    pub total_cost: f64,
    pub duration_ms: u64,
}

pub enum StopReason {
    MaxIterationsReached,
    PerfectScore,
    NoImprovementPlateau { consecutive_failures: u32 },
    UserCancelled,
}
```

## Configuration & Guardrails

```rust
pub struct AutoResearchConfig {
    // Loop bounds
    pub max_iterations: u32,              // default: 10
    pub eval_samples_per_iteration: u32,  // default: 3
    pub plateau_threshold: u32,           // default: 3

    // Quality gates
    pub min_improvement_threshold: f64,   // default: 0.05
    pub min_pass_rate_to_replace: f64,    // default: 0.4
    pub perfect_score_early_stop: bool,   // default: true

    // Model overrides
    pub eval_model: Option<String>,       // None = skill's own model
    pub mutation_model: Option<String>,   // None = skill's own model
    pub skill_execution_model: Option<String>,

    // Safety
    pub max_prompt_length: usize,         // default: 2000
    pub cost_budget: Option<f64>,         // optional USD cap
    pub require_security_scan: bool,      // default: true
}
```

Guardrail logic:
1. Won't replace unless meaningfully better (`min_improvement_threshold`)
2. Won't promote garbage (`min_pass_rate_to_replace`)
3. Plateau detection stops wasting tokens after N fruitless iterations
4. Cost budget prevents runaway spend
5. Security scan on every mutation via `SkillMarketplace::scan_for_injection`
6. Prompt length cap enforced on mutations

## The Loop

```
AutoResearchEngine::run(skill_name)
|
+- 1. Load/generate EvalSuite
|     +- Check TOML for explicit [[eval]] questions
|     +- If none (or partial), auto-generate from skill description via AI
|     +- Merge into hybrid suite
|
+- 2. Baseline eval
|     +- Get current active prompt from PromptEvolver
|     +- Run EvalRunner N times (eval_samples_per_iteration)
|     +- Average pass rates -> baseline_pass_rate
|     +- Record as iteration 0 in history
|
+- 3. Mutation loop (1..=max_iterations)
|     |
|     +- 3a. PromptMutator rewrites the prompt
|     |     +- Input: current best prompt + failing eval questions + sample outputs
|     |     +- Output: mutated prompt text
|     |
|     +- 3b. Safety checks on mutated prompt
|     |     +- Length check (<= max_prompt_length)
|     |     +- Security scan (scan_for_injection)
|     |     +- If fails -> skip iteration, count as plateau tick
|     |
|     +- 3c. Eval the mutated prompt
|     |     +- Run EvalRunner N times
|     |     +- Average pass rates -> candidate_pass_rate
|     |
|     +- 3d. Compare
|     |     +- If candidate > best + min_improvement_threshold:
|     |     |     +- New best! Apply via PromptEvolver::apply_refinement
|     |     |     +- Reset plateau counter
|     |     |     +- Record quality via PromptEvolver::record_quality
|     |     +- Else:
|     |           +- Increment plateau counter
|     |
|     +- 3e. Check stop conditions
|     |     +- perfect_score_early_stop && pass_rate == 1.0 -> stop
|     |     +- plateau_counter >= plateau_threshold -> stop
|     |     +- cost_budget exceeded -> stop
|     |     +- Otherwise -> next iteration
|     |
|     +- 3f. Log iteration to LearningStorage
|
+- 4. Return AutoResearchReport
```

The mutator receives failing eval questions AND sample outputs as context, so it has concrete evidence of what went wrong rather than blindly rewriting.

## AI Interaction Design

### EvalRunner (two calls per sample)

**Call 1 â€” Execute the skill:**
- System prompt: the skill's `prompt_template`
- User message: test input (from skill definition or auto-generated)
- Model: `skill_execution_model` override or skill's own model
- Output: `skill_output`

**Call 2 â€” Judge the output:**
- System prompt: "You are an eval judge. For each question, respond with a JSON array of `{id, passed: bool, reasoning: string}`. Judge ONLY based on the output provided."
- User message: skill output + eval questions
- Model: `eval_model` override (cheap model works here)
- Output: `Vec<EvalResult>`

### PromptMutator (one call)

- System prompt: "You are a prompt engineering expert. Rewrite the given system prompt to improve performance on the failing criteria. Keep the same intent and domain focus. Return ONLY the improved prompt text."
- User message: current prompt + pass rate + failing criteria with reasoning + sample outputs
- Model: `mutation_model` override or skill's own model
- Output: new prompt text

### Auto-generating eval suites (one call)

- System prompt: "Generate 3-6 binary yes/no eval questions for a skill. Focus on: correctness, relevance, format compliance, and safety. Return JSON array of `{id, question, weight}`."
- User message: skill name + description + prompt template
- Output: `Vec<EvalQuestion>`

| `PromptEvolver` | Versioning, `apply_refinement`, `rollback`, `record_quality` |
| `LearningStorage` | Logging iterations via `log_learning` |

No other existing files are modified.

## Testing Strategy

Unit tests per component using `MockExecutor` (same pattern as `skill_authoring.rs`):

| File | Key tests |
|---|---|
| `eval_suite.rs` | Parse from TOML, merge explicit + auto-generated, weight normalization, empty suite rejection |
| `eval_runner.rs` | Pass/fail judgment parsing, averaging across samples, malformed AI response handling, partial pass rates |
| `mutator.rs` | Prompt rewrite produces different text, respects length limits, includes failing criteria in request |
| `config.rs` | Default values, serde roundtrip, budget validation |
| `engine.rs` | Full loop with mocked responses: baseline -> mutation -> improvement -> new best. Plateau stops early. Perfect score stops early. Cost budget stops early. Security scan rejects bad mutation. |

Estimated: ~30-35 new tests. Each file stays well under 1000 lines (no rustc stack overflow risk).

## Design Decisions

1. **Hybrid evals** â€” Skills can define explicit eval questions, but the system auto-generates them when missing. Works out of the box for every skill.
2. **Autonomous with guardrails** â€” Runs without human intervention but has safety rails (min improvement threshold, plateau detection, cost budget, security scan, rollback).
3. **Configurable models** â€” Sensible defaults (use skill's own model) with overrides for cost optimization. Fits existing `hive_ai` routing patterns.
4. **Separate eval execution from judgment** â€” Call 1 tests the skill realistically, Call 2 is pure binary classification (ideal for cheap models).
5. **Evidence-based mutation** â€” The mutator sees failing criteria AND sample outputs, enabling targeted improvements rather than random rewrites.
6. **Full version history** â€” Every winning mutation creates a new `PromptEvolver` version. Rollback to any prior version at any time.
